# 前端信息架构与页面职责

Last updated: 2026-06-03.

本文记录当前前端的页面分工、菜单结构、同步页状态语义和颜色规范。目标是避免不同页面互相复刻，减少“看起来都像工作台”的信息重复。

## 总体原则

- 页面按投研流水线组织，而不是按技术模块堆叠。
- 首页只做投研决策，不承载重型操作；运维页只做工程排障，不解释策略结果。
- 数据查看和数据同步分离，避免查询表格、执行参数和长任务进度挤在同一屏。
- 因子定义和因子评估分离，表达式编辑只在创建/编辑时打开；日常评估优先看缓存、报告和组合结果。
- 右侧上下文面板随当前页面变化，展示页面最需要的少量状态，不复制主页面所有表格。

## 路由与职责

| 路由 | 页面 | 职责 |
|---|---|---|
| `/home` | 今日投研工作台 | Research Readiness、今日行动建议、投研输入口径、流水线推进和影响研究的近期事件。 |
| `/data` | 数据查看 | 查看日线、分钟线、基础股票、财务、指标、概念和舆情的最新口径；不触发同步。 |
| `/data/sync` | 数据同步 | 同步任务目录、预设方案、执行参数、队列、运行进度、Relay Key 和同步日志。 |
| `/explorer` | 数据浏览器 | Parquet/DuckDB 表结构与预览；未知行数显示“未统计”，精确行数显式触发。 |
| `/watchlist` | 自选股 | 研究股票池分组和个股详情入口。 |
| `/factor` | 因子定义 | 因子目录、覆盖率、参数版本和预计算入口；表达式创建/编辑时才展开。 |
| `/factor/detail/:factorName` | 因子详情 | 因子公式、人话解释、覆盖率、缓存和研究结果。 |
| `/factor/evaluation` | 因子评估 | IC、ICIR、多空收益、回撤、换手和已计算组合；只消费预计算缓存。 |
| `/research` | 研究实验室 | 研究假设、外部链接、Obsidian/本地知识库衔接、证据链、实验记录和失败复盘。 |
| `/backtest` | 策略回测 | 代码编辑、参数配置、运行日志、回测报告和优化入口。 |
| `/trade` | 模拟 / 实盘 | 信号、账户、订单预览、真实下单二次确认和风控护栏。 |
| `/monitor` | 系统运维控制台 | 服务拓扑、值班排障清单、运行控制、任务表、存储巡检和同步审计日志。 |
| `/docs` | 文档中心 | 用户手册、数据源、因子、AKQuant、运维说明等入口。 |

## 工作台 vs 系统运维

| 页面 | 使用者视角 | 首屏应该回答 | 不应该做 |
|---|---|---|---|
| `/home` 工作台 | 投研负责人 / 策略研究者 | 今天先补数据、做研究、跑回测，还是检查交易护栏？ | 展开服务拓扑、进程、端口、错误日志细节。 |
| `/monitor` 系统运维 | 数据工程 / SRE / 值班者 | 哪个服务、队列、存储或交易护栏需要立即处理？ | 判断因子是否可交易、给策略研究建议。 |

工作台的核心组件：

- `Research Readiness`：研究就绪度，用数据口径、同步影响、最近回测和交易护栏综合判断。
- `今日行动建议`：把异常或缺口转成下一步动作，例如“看数据”“查看同步”“去回测”“检查交易”。
- `投研输入口径`：只展示对研究决策有影响的数据最新日期。
- `投研流水线推进`：数据、研究、因子定义、因子评估、回测、交易的跳转入口。

系统运维的核心组件：

- `服务拓扑`：FastAPI、Sync Worker、Storage、QMT/Quote、Order Guard。
- `值班排障清单`：把不可用服务、失败任务、队列堆积、存储缺口、真实下单开关转为排障项。
- `运行控制面板`：同步入口、当前同步、后端排队、活动任务、失败任务、存储异常。
- `任务表 / 存储巡检 / 同步审计日志`：面向工程排错的明细视图。

## 数据同步状态语义

同步状态来自 `GET /api/data/sync/status`：

| 字段 | 前端解释 |
|---|---|
| `status` | 当前任务状态：`idle`、`queued`、`running`、`completed`、`failed`、`cancelled`。 |
| `sync_service_available` | 同步服务是否可用。不可用时禁用提交入口。 |
| `can_trigger` | 服务是否接受新提交。它不等于“当前没有任务运行”。 |
| `details.queue_mode` | 为 `true` 时运行中仍可提交，新任务进入后端队列。 |
| `details.queue_pending_count` | 后端队列待执行数量。 |
| `reason` | 不可提交或服务不可用时的解释。 |

前端展示规则：

- 右侧上下文显示“提交入口 / 运行说明”，不再显示“允许触发=是/否”。
- `status=running` 时，提示当前执行任务和排队数量。
- 顶部“一键同步核心数据”在 `queued/running` 时禁用，避免核心任务被重复提交。

## 颜色语义

| 场景 | 红色 | 绿色 |
|---|---|---|
| 平台状态 | 需要关注、异常、失败、真实下单开启等风险状态 | 正常、就绪、可用、安全状态 |
| A 股行情 | 上涨 | 下跌 |

不要用同一套涨跌颜色解释平台健康状态。平台状态与行情涨跌是两个语义域。

## 开发注意事项

- 新增页面需同步更新 `frontend/src/router/index.ts`、`frontend/src/app/navigation.ts` 和本文档。
- 页面使用 `usePageContext` 时，右侧上下文应提供该页面的决策摘要，不复制主表格。
- 修改前端后至少运行 `cd frontend; npm run build`。
- 如果改了同步页与同步状态契约，同时更新 `frontend/src/api/sync.ts`、相关后端测试和本文“数据同步状态语义”。
